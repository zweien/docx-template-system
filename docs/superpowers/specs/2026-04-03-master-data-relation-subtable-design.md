# 主数据关系子表格字段设计

## 背景

当前主数据模块已经支持基础字段、单向关联字段、记录表格视图和 Excel 导入导出，但和 Airtable 在“复杂关系建模”上仍有明显差距。

本期目标不是补齐所有 Airtable 能力，而是优先实现一套可复用的“关系子表格字段”能力，支撑论文-作者、项目-成员、订单-商品明细、合同-付款节点等通用场景。

## 目标

- 支持一对一、一对多、多对多关系建模。
- 支持反向字段自动生成，且正反两侧都可编辑并保持同步。
- 支持每个关系字段独立配置一组边属性子字段。
- 记录编辑页中，关系字段以“内嵌子表格”方式编辑。
- 底层允许新增隐藏的关系实体/关系边存储结构，用户不直接感知中间表。
- 导入支持“主表嵌套导入”和“关系明细表导入”两种模式。
- 支持表级业务唯一键配置，用于导入时稳定匹配主表记录和关联表记录。

## 非目标

- 本期不实现 Lookup、Rollup、Formula。
- 本期不新增看板、日历等视图形态。
- 本期不实现单元格级评论、协同编辑、历史版本审计 UI。
- 本期不追求复杂关系子表格的批量编辑体验，优先保证模型正确、读写一致、导入可用。

## 核心抽象

### 1. 关系子表格字段

在字段体系中新增一种“关系子表格字段”抽象。它不是论文场景专用能力，而是任意主表到关联表的通用关系字段。

每个关系子表格字段需要表达以下元数据：

- 关联目标表。
- 本侧关系基数：单条或多条。
- 反向侧关系基数：创建正向字段时显式指定，用于区分一对一和一对多；该值落到自动生成的反向字段自身 `relationCardinality` 上，不需要额外数据库列。
- 反向字段绑定关系。
- 反向字段默认自动命名，后续允许改字段名和显示名。
- 关联记录展示字段。
- 该关系上的边属性子字段 schema。

关系基数规则：

- 一对一：本侧单条 + 反向侧单条。
- 一对多：本侧单条 + 反向侧多条。
- 多对多：本侧多条 + 反向侧自动生成多条。

结构性约束：

- 反向字段自动生成。
- 字段名/显示名可改。
- 目标表、关系基数、反向字段绑定关系不可单侧随意修改。
- 如需调整结构性配置，必须通过“重建关系字段对”这类受控流程完成。

### 2. 边属性子字段

每个关系字段都可以配置一组边属性子字段，字段类型复用现有主数据字段体系，例如 `TEXT`、`NUMBER`、`DATE`、`SELECT`、`MULTISELECT`、`EMAIL`、`PHONE`、`FILE`。

示例：

- 论文-作者关系：作者顺序、是否通讯作者、是否共同一作、署名单位、贡献说明。
- 项目-成员关系：项目角色、投入比例、开始日期、是否负责人。
- 订单-商品关系：数量、单价、折扣、备注。

这些属性属于“关系行”本身，不属于主表记录，也不属于被关联表记录。

### 3. 隐藏关系实体表

对于带边属性的关系，底层不建议只用一张普通边表硬扛所有场景，而是采用“系统隐藏的关系实体表”思路：

- 用户在 UI 上只看到主表记录里的关系子表格字段。
- 系统在底层为该关系字段对托管一组关系行记录，用来承载“主记录、关联记录、边属性、顺序”等信息。
- 这张关系实体表对业务用户隐藏，但服务层可以统一查询、校验、导入、同步反向字段。

为了兼容现有表格展示、表单回填、导出和 AI 查询，`DataRecord.data` 中可以保留关系字段的读模型快照；但关系一致性的真相来源应切到隐藏关系实体存储，而不是继续依赖 JSONB 单点维护。

## 推荐方案

采用“隐藏关系实体表 + JSONB 读模型快照”的混合方案。

### 为什么不选纯 JSONB

继续只在 `DataRecord.data` 里维护关系，会让双向同步、多对多约束、删除清理和后续 Lookup/Rollup 都变得脆弱，应用层容易产生不一致数据。

### 为什么不选纯隐藏关系表且完全移除 JSONB 快照

如果关系字段完全不落 JSONB 快照，当前记录表格渲染、表单默认值、导出、AI 查询路径都要大改，第一期实施风险偏高。

### 为什么选混合方案

- 写入一致性以隐藏关系实体存储为准。
- JSONB 关系快照只承担读优化和兼容旧 UI 的职责。
- 后续做 Lookup/Rollup/Formula 时，可以直接基于关系实体查询扩展，不必再反复修补 JSONB 双写逻辑。

## 数据模型设计

### DataField 扩展

建议在 `DataField` 上补充关系结构元数据：

- `relationCardinality`：`SINGLE` 或 `MULTIPLE`。
- `inverseFieldId`：反向字段 ID。
- `isSystemManagedInverse`：标记该字段是否由系统自动生成。
- `relationSchema`：边属性子字段 schema。
- `isBusinessKey` 或表级唯一键配置引用：用于导入匹配。

其中 `relationTo`、`displayField` 可以继续保留，但需要明确它们属于关系字段结构性配置。

### 隐藏关系实体存储

建议新增一套关系行存储模型，逻辑上至少包含：

- 关系字段对标识。
- 主记录 ID。
- 关联记录 ID。
- 边属性数据。
- 行顺序。
- 创建/更新时间。

如果实现层选择“系统托管关系实体表”而不是“单一通用边表”，也要保证服务层抽象对上层统一暴露为“按关系字段读写关系行”，避免 UI 和业务层感知底层存储细节。

### DataRecord.data 中的关系快照

关系字段在 JSONB 里的快照建议分两种形态：

- 单条关系：保存一个对象快照，包含关联记录 ID、展示值、边属性摘要。
- 多条关系：保存对象数组快照，每项包含关联记录 ID、展示值、边属性摘要、排序值。

该快照只用于列表展示、表单初始回填、导出默认值，不作为关系写入真相来源。

## 写入链路设计

### 字段创建/更新

创建关系子表格字段时：

1. 在本表创建正向字段。
2. 在目标表自动创建反向字段。
3. 建立 `inverseFieldId` 互相绑定。
4. 正向字段创建请求必须携带反向侧基数；系统据此生成反向字段的 `relationCardinality`，从而可明确表达一对一、一对多、多对多。
5. 初始化该关系字段对的边属性 schema。

更新字段时：

- 允许修改字段名、显示名、展示字段、边属性子字段定义中的非破坏性内容。
- 不允许单侧直接修改目标表、基数、反向绑定。
- 对边属性 schema 的破坏性变更需要有明确迁移策略，例如新增字段允许，删除/改类型需要校验现有关系行数据。

### 记录创建/编辑

保存记录时，普通字段和关系字段要分离处理：

1. 普通字段按现有字段校验和 JSONB 写入流程处理。
2. 关系子表格字段先从表单提交值解析出关系行集合。
3. 读取该记录旧的关系行集合，计算新增、更新、删除 diff。
4. 在数据库事务中写入隐藏关系实体存储。
5. 同事务刷新本记录 JSONB 关系快照。
6. 同事务刷新所有受影响对端记录的反向字段 JSONB 快照。

单值约束校验：

- 本侧单条关系字段对同一条主记录最多只能保留 1 条关系行。
- 如果反向字段也是单条，则同一条对端记录最多只能被该字段对中的 1 条关系行占用。
- 违反时返回明确冲突错误，不做部分写入。

### 删除记录

删除记录时：

1. 找出所有和该记录相关的关系行。
2. 删除这些关系行。
3. 批量刷新对端记录 JSONB 反向快照，等价于自动解除引用。
4. 删除记录本身。

本期默认采用“自动解除引用并保留对端记录”，不做“禁止删除被引用记录”的约束策略。

## UI 设计

### 字段配置页

在字段配置表单中，关系子表格字段需要增加：

- 目标表选择。
- 本侧基数选择：单条或多条。
- 反向侧基数选择：单条或多条；当本侧为多条时，反向侧只能为多条。
- 展示字段选择。
- 反向字段自动生成提示和默认命名预览。
- 边属性子字段配置区，可新增、编辑、删除、排序子字段。

交互约束：

- 新建字段时允许配置结构性参数。
- 已创建并绑定反向字段后，目标表、基数、反向字段绑定关系应锁定。
- 反向字段显示为系统自动生成字段，但允许改名。

### 记录编辑页

关系子表格字段统一渲染为“表单中的一个字段区块”，不是单独业务特例页签。

字段区块内是一张内嵌子表格：

- 每行选择一条关联记录。
- 每行可编辑该关系上的边属性子字段。
- 支持新增行、删除行、调整行顺序。
- 单条关系字段最多保留一行，多条关系字段可保留多行。
- 反向字段侧也使用同一套组件，保证双向可写。

### 列表页展示

记录表格中，关系字段优先展示关联记录的展示值：

- 单条关系显示一个 badge 或文本。
- 多条关系显示多个 badge，超出数量可折叠。
- 边属性默认不在主列表展开，优先在记录编辑页或详情区查看，避免列表过度拥挤。

## 导入设计

### 表级业务唯一键

每张主数据表需要支持配置 1 个或多个业务唯一键字段，用于导入时稳定匹配记录。

约束建议：

- 业务唯一键字段必须来自该表自身字段。
- 导入匹配时使用“全部唯一键字段共同匹配”的组合键语义。
- 如果命中多条记录，视为数据异常，预览阶段直接报错。
- 如果未命中记录，则根据导入策略选择新建或报错。

### 主表嵌套导入

适用于用户只想导入主表，但其中某列包含关系子表格明细。

支持两类输入格式：

- 嵌套 JSON 列：一列中直接放关系行数组，每行包含关联记录唯一键和值属性。
- 约定分隔格式列：适用于轻量场景，但能力弱于 JSON，优先级可低于 JSON。

导入流程：

1. 先按主表字段映射解析普通字段。
2. 对关系子表格字段，解析嵌套关系行。
3. 按被关联表业务唯一键匹配或创建关联记录。
4. 写入主记录和关系行。
5. 刷新双向 JSONB 快照。

### 关系明细表导入

适用于用户已有主表、关联表、中间表三张 Excel，且希望把中间表直接导入到某个关系子表格字段。

导入向导需要支持：

- 选择目标主表。
- 选择目标关系子表格字段。
- 映射主表唯一键列。
- 映射关联表唯一键列。
- 映射边属性子字段列。

每一行导入数据表示一条关系行。写入时按主表唯一键定位主记录，按关联表唯一键定位关联记录，再写入关系行和边属性。

### 导入预览校验

预览阶段必须拦截以下问题：

- 主表唯一键缺失。
- 关联表唯一键缺失。
- 唯一键重复命中多条记录。
- 单条关系字段出现多行关系数据。
- 必填边属性缺失。
- 边属性类型不合法。

只有预览校验通过后才允许正式写入，避免批量脏数据进入关系存储。

## 服务与 API 拆分

建议按职责拆分服务层，避免继续把复杂逻辑堆进 `data-record.service.ts`：

- `data-field.service.ts`：字段保存、关系字段对创建、反向字段绑定、结构性校验、边属性 schema 维护。
- `data-relation.service.ts`：关系行 diff、隐藏关系实体写入、双向快照刷新、删除记录时关系清理。
- `data-record.service.ts`：普通字段校验、记录事务保存，协调 relation service。
- `data-table.service.ts`：表详情读取、字段聚合、业务唯一键配置读取。
- `data-import.service.ts`：主表嵌套导入、关系明细表导入、唯一键匹配、预览校验、批量 upsert。

API 层继续保持薄封装：

- 复用现有字段保存、记录保存、导入接口风格。
- 仅为“关系记录搜索选择器”“关系明细表导入预览/提交”“表级业务唯一键配置”新增必要的窄接口。

## 迁移策略

现有 `RELATION` 字段目前是单值关系，且关系值直接存储在 `DataRecord.data[fieldKey]` 中。

迁移建议：

1. 扩展字段模型和关系存储模型。
2. 为现有 `RELATION` 字段生成反向字段和关系字段对绑定信息。
3. 扫描历史 JSONB 关系值，回填隐藏关系实体存储。
4. 按新规则刷新正反两侧 JSONB 快照。
5. 对历史脏数据做迁移前报告：
   - 引用不存在的记录。
   - 单值字段出现数组或重复关系。
   - 目标表/展示字段配置缺失。
6. 对严重冲突默认阻断迁移，输出报告后人工处理，不做静默截断。

## 测试策略

### 服务层集成测试

- 一对一创建、修改、删除、反向同步。
- 一对多创建、修改、删除、反向同步。
- 多对多创建、修改、删除、反向同步。
- 单值约束冲突。
- 边属性新增、编辑、删除、排序。
- 删除记录后自动解除对端引用。
- 主表嵌套导入。
- 关系明细表导入。
- 业务唯一键重复命中和未命中策略。
- 迁移脚本回填与脏数据报告。

### 组件测试

- 字段配置表单中的关系子表格字段配置、反向字段自动生成提示、结构性字段锁定。
- 记录编辑页内嵌关系子表格的增删行、选择关联记录、编辑边属性、单值行数限制。
- 列表页关系字段展示。
- 导入映射向导对两种导入模式的字段映射和错误提示。

### 回归测试

- 现有普通字段类型创建、编辑、列表展示不受影响。
- 现有模板关联主数据、批量生成、AI 数据查询路径不因关系字段改造出现回归。
- 现有视图筛选/排序/隐藏字段能力在非关系字段场景下保持可用。

## 风险与约束

- 这是一个数据模型级改造，不能只做 UI 补丁；必须优先保证写入事务一致性和迁移可回滚。
- 如果边属性 schema 直接塞进 `DataField` JSON，需要明确校验和版本演进策略，否则后续改 schema 容易产生历史关系行解释歧义。
- 如果隐藏关系实体存储抽象设计不清晰，后续 Lookup/Rollup 会继续叠技术债；因此服务层边界必须先拆出来。
- 导入能力范围较大，建议实现时先优先打通“业务唯一键 + 关系明细表导入”，再补“主表嵌套 JSON 导入”，避免一次性把解析器复杂度拉满。

## 建议实施顺序

1. 字段模型扩展 + 隐藏关系实体存储模型。
2. 关系字段对创建/反向字段自动生成/结构性锁定。
3. 记录保存链路接入关系行 diff 和双向快照刷新。
4. 记录编辑页内嵌关系子表格组件。
5. 列表页关系字段展示适配。
6. 表级业务唯一键配置。
7. 关系明细表导入。
8. 主表嵌套关系导入。
9. 历史数据迁移脚本和回归测试。

## 实现决策

- 隐藏关系实体存储第一期采用“单一通用关系行表”。这样 Prisma schema 更稳定，服务层可以先围绕统一关系行模型建立读写、校验、导入和迁移逻辑；如果后续确实需要物理拆分，再在服务层内部演进，不把复杂度暴露给 UI。
- 业务唯一键配置放在表级独立元数据上，保存“该表由哪些字段组成唯一键”以及字段顺序；`DataField` 只负责描述字段本身，不把组合唯一键语义分散塞到字段定义里，避免职责混杂。
- 边属性子字段第一期不允许再嵌套 `RELATION` 字段，只支持非关系型字段。这样可以先把“主记录-关联记录-边属性”三层模型做稳，避免一开始就引入递归关系编辑和循环依赖校验。
