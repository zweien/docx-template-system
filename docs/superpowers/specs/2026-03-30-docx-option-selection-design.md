# DOCX 选项勾选生成设计

**日期**: 2026-03-30
**状态**: 设计完成，等待用户审阅

## 1. 目标

为 docx 模板增加“单选 / 多选”能力，使模板作者可以在 Word 中直接声明选项组，用户在系统里完成选择后，生成文档时仅替换选项前的勾选符号：

- 选中: `☑`
- 未选中: `☐`

核心目标：

- 保留 Word 中原有文字内容和格式
- 不重写选项文本
- 第一版仅支持单份填写页 `templates/[id]/fill`

## 2. 背景

当前系统支持：

- 普通占位符 `{{key}}`
- 明细表块 `{{#name}} ... {{/name}}`

但不支持模板中常见的“勾选项”场景，例如：

- 性别单选
- 证件类型单选
- 爱好多选
- 材料清单多选

用户提供的需求有几个硬约束：

1. 采用一套新的模板语法
2. 语法尽量保留 docx 中原有文字和格式
3. 支持单选和多选
4. 填写页统一使用弹层选择器
5. 生成时只替换勾选框符号，不改后面的文字
6. 第一版只支持单份填写，不做批量生成

## 3. 设计结论

### 3.1 推荐方案

采用“控制行 + 选项项”的混合语法。

- 控制信息单独存在，用于声明字段 key 和单选/多选类型
- 选项项仍然保留为普通 Word 文本内容
- 必要时允许在选项项上带轻量辅助标记，用于降低解析歧义
- 控制信息在 Word 中可见，模板作者可以直接编辑

该方案的理由：

- 比纯自然文本识别更稳定，不靠猜
- 比块语法侵入性更低，更接近 Word 的日常编辑方式
- 最容易满足“只改框，不改字”的生成要求

### 3.2 第一版范围

包含：

- 模板解析时识别选项组
- 模板模型中保存选项组配置
- 填写页渲染单选 / 多选弹层字段
- 生成 docx 时只替换勾选框符号
- 模板语法错误时明确报错

不包含：

- 批量生成链路支持
- 自动识别自然语言选项
- 隐藏控制信息
- 跨表格 / 页眉页脚 / 文本框等复杂区域解析
- 容错修复模板语法

## 4. 模板语法

### 4.1 语法原则

- 控制信息可见
- 选项文字保持普通 Word 内容
- 系统只识别最小必要标记
- 不要求模板作者在 Word 中编写大段“像代码”的块语法

### 4.2 建议语法形态

第一版建议采用一条可见控制行 + 多条选项项：

```text
{{选项:性别|single}}
□ 男
□ 女
```

多选示例：

```text
{{选项:爱好|multiple}}
□ 篮球
□ 足球
□ 音乐
```

说明：

- `性别`、`爱好` 是字段 key
- `single` 表示单选
- `multiple` 表示多选
- 选项项前的 `□` 是生成阶段的主要替换目标

当前实现兼容一类真实 Word 模板常见的内联写法，但类型仍需通过前置控制行显式声明：

```text
{{选项:单项|single}}
单项：☑是 ☐否

{{选项:多选|multiple}}
多选：☐选项1 ☑选项2 ☑选项3 ☐选项4
```

这类写法在 docx XML 中通常不是普通 `□` 字符，而是 `w:sym` + `Wingdings 2` 符号。实现层会把它们识别为：

- `单项` -> `single`
- `多选` -> `multiple`

并保留后面的选项文字作为 `label` / `value`。

### 4.3 规则

- 一个选项组必须有且仅有一条控制行
- 选项项必须紧跟控制行附近出现
- 第一版只支持“选项前带独立勾选框符号”的格式
- 选项文字本身不参与模板语法解析，只作为普通展示文本保留
- 控制行不出现在最终生成文档中

对于内联 `w:sym` 形态，补充规则：

- 必须有前置控制行显式声明字段 key 和 `single` / `multiple`
- 控制行后的下一个可见段落中，必须能识别出至少两个选项
- 勾选框必须是当前支持的 `Wingdings 2` 符号
- 选项文本即使被 Word 拆成多个连续 run，例如 `选项` + `1`，也需要在解析阶段拼回完整标签
- 生成时只更新对应 `w:sym` 的字符值，不重写后续文本 run

### 4.4 非目标

第一版不支持以下写法：

- 同一个字段分散在多个不连续区域
- 未声明控制行，仅靠 `□ 男 / □ 女` 自动猜测
- 控制行跨多个段落
- 复杂嵌套结构
- 任意自然语言自动推断字段 key 或单选/多选类型

## 5. 数据模型设计

### 5.1 Placeholder 类型扩展

当前 `PlaceholderItem.inputType` 主要是：

- `TEXT`
- `TEXTAREA`
- `TABLE`

本功能建议新增两类：

- `CHOICE_SINGLE`
- `CHOICE_MULTI`

### 5.2 选项配置结构

建议为选项型占位符增加结构化配置：

```typescript
interface ChoiceOption {
  value: string;
  label: string;
}

interface ChoicePlaceholderConfig {
  mode: "single" | "multiple";
  options: ChoiceOption[];
  marker: {
    checked: "☑";
    unchecked: "☐";
    template: "□";
  };
}
```

其中：

- `label` 对应文档中实际显示文本
- `value` 用于前端存储和提交
- `marker.template` 表示模板中原始勾选框符号

### 5.3 填写数据结构

```typescript
type FormDataValue =
  | string
  | string[]
  | Array<Record<string, string>>;
```

约定：

- 单选值存为 `string`
- 多选值存为 `string[]`

示例：

```typescript
{
  性别: "男",
  爱好: ["篮球", "音乐"]
}
```

## 6. 解析设计

### 6.1 解析阶段新增能力

模板上传后，解析器在现有普通占位符和表格块解析之外，新增“选项组解析”。

解析结果应包含：

- 字段 key
- 选项组类型 `single` / `multiple`
- 选项列表
- 每个选项在文档中的定位信息

解析输入来源当前有两类：

- 可见控制行 + 独立选项项
- 内联 `w:sym` Word 勾选段落

### 6.2 为什么必须存定位信息

本需求的关键不是识别选项，而是“生成时只改框不改字”。

因此生成阶段不能简单整段替换。解析阶段必须提前拿到足够稳定的定位信息，用于后续只修改选项前的勾选框字符。

### 6.3 定位信息要求

第一版建议至少记录：

- 所在段落顺序
- 选项在段落中的相对位置
- 勾选框字符对应的 run 或文本片段位置
- 对于内联 `w:sym` 形态，还需要能定位到对应符号节点

这里的目标不是做一个通用 Word 编辑器，只要足够支撑：

- 找到某组选项
- 找到每个选项前的框
- 安全地把 `□` 替换成 `☑` 或 `☐`

### 6.4 解析失败策略

以下情况直接报错，不允许静默跳过：

- 控制行缺失
- 控制行重复
- 选项项为空
- 同一组选项区域格式不合法
- 勾选框符号不符合第一版约束

补充：

- 如果是内联 `w:sym` 结构，但同一段落里无法稳定拼出完整选项标签，也视为格式不合法
- 如果模板使用了别的符号字体而不是当前支持的 `Wingdings 2` 勾选框，也按不支持处理

## 7. 前端交互设计

### 7.1 填写页表现

单选和多选统一使用“弹层里勾选”的交互。

表现形式：

- 字段在表单中显示为一个可点击输入区
- 点击后弹出选项面板
- 单选时仅允许选择一个值
- 多选时允许选择多个值

### 7.2 字段展示

建议规则：

- 未选择时显示占位提示
- 单选显示当前选中的 `label`
- 多选显示已选项数量或标签摘要

### 7.3 为什么统一交互

这样可以降低认知成本：

- 用户只记住一种操作模式
- UI 组件边界清晰
- 后续如果扩展搜索、分组、禁用项，也更容易统一实现

## 8. 生成设计

### 8.1 生成原则

生成 docx 时：

- 不重写整个选项段落
- 不替换选项文字
- 只替换选项前的勾选框字符

### 8.2 生成行为

单选示例：

模板：

```text
{{选项:性别|single}}
□ 男
□ 女
```

用户选择“男”后：

```text
☑ 男
☐ 女
```

多选示例：

```text
{{选项:爱好|multiple}}
□ 篮球
□ 足球
□ 音乐
```

用户选择“篮球”“音乐”后：

```text
☑ 篮球
☐ 足球
☑ 音乐
```

### 8.3 文本与格式保留

必须保留：

- 选项文字内容
- 字号、字体、颜色
- 加粗、斜体、下划线
- 段落缩进、对齐、换行

也就是说，系统不是“重新生成一组选项行”，而是在模板已有内容上做最小变更。

## 9. 错误处理

### 9.1 模板阶段

模板不合法时，上传或解析阶段直接失败，并返回明确错误信息：

- 哪个字段有问题
- 是哪一类语法错误
- 应该如何修正

### 9.2 填写阶段

如果用户提交的值不符合约束：

- 单选传多个值
- 多选传非法值
- 传入了模板中不存在的选项

则阻止生成，并返回表单校验错误。

### 9.3 生成阶段

如果定位信息与模板内容不一致：

- 直接报错
- 不输出半正确文档

第一版不允许“部分字段替换成功，部分字段跳过”。

## 12. 风险与规避

| 风险 | 说明 | 规避方式 |
|------|------|----------|
| Word run 拆分导致定位不稳 | 勾选框和文本可能被拆分到多个 run | 解析阶段按段落 / run 做显式定位，不靠整段字符串替换 |
| 模板作者写法不统一 | 可见语法容易被自由发挥 | 第一版严格限制写法，模板错误直接报错 |
| 只改框不改字的实现复杂 | 比整段重写更难 | 第一版缩小范围，只支持明确格式 |
| 后续批量生成扩展成本 | 单份先做，后续要复用 | 在 placeholder 配置和生成接口层保持结构化设计 |

## 13. 实施建议

建议按以下顺序实施：

1. 先确定最终控制行语法细节
2. 改造解析器，能稳定识别选项组
3. 扩展 placeholder 数据模型
4. 实现填写页弹层组件
5. 实现生成阶段勾选框替换
6. 补齐解析、生成、集成测试

第一阶段的成功标准很简单：

- 能解析一套明确语法
- 能在单份填写页让用户选值
- 能只替换勾选框
- 模板出错时能明确报错
